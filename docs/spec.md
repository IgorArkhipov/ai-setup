Spec is a "How Exactly and When It's Done" document.
Spec represents the solution design.
It is a document an agent can use to write code without additional clarification.

How it is created: a spec is well generated by an agent from a brief - an agent can structure requirements, suggest acceptance criteria, and describe error scenarios.
But a spec requires human decisions: choosing an approach from alternatives, defining the scope (what is in and out), setting implementation constraints, and prioritizing requirements.
The agent proposes - the human decides.

It answers the following questions:
- What is in scope and what is not? (solution boundaries)
- What are the specific requirements? (system behavior)
- What invariants must not be broken? (constraints)
- What are the acceptance criteria? (when it is "done")
- What are the implementation constraints? (what must not be changed, which patterns to follow)

Now you are a strict spec reviewer for AI agents. Review the provided spec using the TAUS criteria.

For each criterion, give a pass/fail assessment:

1. Testable - are there specific acceptance criteria that can be used to write an automated test?
2. Ambiguous-free - are there ambiguous words ("fast", "convenient", "if needed", "etc.")?
3. Uniform - are all states described (loading, error, success, empty)? Are all error scenarios covered?
4. Scoped - is this one feature? Is it under 1500 words? Does it affect no more than 3 modules?

Additionally, check:
5. Is there a link to the Brief/Issue?
6. Is the scope stated clearly (what is in AND what is NOT in scope)?
7. Are the invariants listed?
8. Are the implementation constraints specified?

For each fail:
- Quote from the spec
- Why this is a problem for the agent
- A concrete suggestion for how to fix it

If all criteria pass, write: "0 issues, the spec is ready for implementation."
